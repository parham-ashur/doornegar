## 1. Security

### Verified state
- `.env`, `.env.backup`, `.env.vercel` are **not** in git history (verified via `git ls-files`) — they live only on Parham's laptop. Severity of secret-exposure findings downgraded accordingly.
- `.gitignore` covers `.env` / `.env.local` / `.env.production` but **not** `.env.backup` or `.env.vercel` — latent risk from a future `git add .`.
- Dashboard client-side password confirmed at `frontend/src/app/[locale]/dashboard/page.tsx:110` — ships in the JS bundle.

### Blockers
| # | Finding | Location | Action |
|---|---|---|---|
| S1 | Dashboard admin password hardcoded client-side (`doornegar2026`) | `frontend/src/app/[locale]/dashboard/page.tsx:110,447` | Remove `ADMIN_PASS`; rely only on backend `ADMIN_TOKEN` sent as `Authorization: Bearer` |
| S2 | `.gitignore` does not cover `.env.*` variants; `.env.backup` & `.env.vercel` are on disk untracked but one `git add .` from being committed | `.gitignore`, `backend/.env.backup`, `frontend/.env.vercel` | Tighten ignore to `.env*` with `!.env.example`; delete the backup; delete the Vercel OIDC file |
| S3 | `POST /api/v1/social/channels` is unauthenticated — anyone can add tracked Telegram channels | `backend/app/api/v1/social.py:38-55` | Add `Depends(require_admin)` |
| S4 | CORS allows stale Vercel preview `frontend-tau-six-36.vercel.app` | Railway env `CORS_ORIGINS` | Lock to `doornegar.org`, `www.doornegar.org`, `localhost:3000` |

### Risks
| # | Finding | Location | Action |
|---|---|---|---|
| S5 | Admin token kept in `localStorage` (XSS-extractable) | `frontend/src/app/[locale]/dashboard/page.tsx:126` | Move to `HttpOnly; Secure; SameSite=Strict` cookie from backend |
| S6 | Admin dashboard endpoints not rate-limited; no lockout on bad password | `backend/app/api/v1/admin.py` | Apply `slowapi` limiter (e.g. 5/min) to `/api/v1/admin/*` |
| S7 | LLM keys and DB creds exist in plaintext on laptop; no rotation playbook | `backend/.env`, `backend/.env.backup` | Document a rotation runbook in project-management; rotate on any laptop compromise |
| S8 | No IP-redaction policy documented for logs (matters for Iranian users) | Railway logs, Cloudflare analytics | Confirm Cloudflare analytics is anonymized; add `X-Redact-IP` policy doc |

### Nice-to-have
- Dependencies are modern (Next 14.2, FastAPI >=0.115, SQLAlchemy 2). Monthly `npm audit` and SDK-version check is enough.
- No raw SQL touching user input — ORM usage is safe.
- Docker Compose dev password (`doornegar_dev`) is intentional and bound to localhost — fine.
